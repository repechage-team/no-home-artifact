## 1. 서론

AI에게 단순히 “이거 만들어줘”, “이거 추가해줘”라고 마구잡이로 명령을 내리면 AI는 요구사항이 누락되거나, 기존 구조와 어긋나거나, 검증되지 않은 코드를 생성할 수 있습니다.

그러나 이 문제를 방지하기 위해 프로젝트의 모든 가이드라인을 다 잡고 오차 없는 완벽한 계획을 세워서 AI에게 시킬 순 없습니다.  그 이유는 다음과 같습니다.

1. 사람이 프로그램의 모든 변수를 다 잡고 가이드라인을 세울 순 없습니다.
2. 완벽한 계획을 세우는 동안 AI의 빠른 생산성을 제대로 활용하지 못합니다.

그렇다면 우리는 어떻게 AI를 써야할까요? 

이 글에는 AI에게 모든 것을 맡기는 것이 아니라, 작은 작업 단위와 명확한 검증 구조 안에서 AI의 속도를 활용하는 방법에 대한 저의 생각들을 담았습니다.

## 2. AI의 장단점 & 이를 어떻게 활용하고 제어할 것인가

### 장점

- `빠른 생산성`→  Agile한 개발 프로세스를 진행할 때  이점을 가짐.
    - 왜 그럴까?
        - AI의 가장 큰 장점은 완성도 높은 최종 제품을 한 번에 만드는 능력보다, 작은 기능 단위를 빠르게 구현해 피드백 루프를 짧게 만드는 데 있습니다. 따라서 AI는 대규모 설계를 한 번에 맡기기보다, MVP나 컴포넌트 단위의 실험에 투입할 때 효과가 큽니다.
    - Agile의 핵심: `짧은 주기의 개발(Sprint) → 빠른 출시 → 피드백`을 통한 지속적인 개선
        - AI는 이 사이클에서 발생하는 시간적 병목을 줄여줍니다.
    - `어떻게 활용할까?`
        - 작업 단위를 컴포넌트로 나누어서 Agile한 개발 프로세스를 도입
        - 이 과정에서 계획과 실행을 분리하는 워크플로우가 필요

### 단점

1. `여러 작업을 동시에 시켰을 때 주의력 분산`
    - AI에게 요청할 때 “너는 10년차 시니어 백엔드 개발자야”라는 식으로 페르소나를 부여하는 프롬포트를 본 적이 있을 것입니다.
        - 이렇게 페르소나를 부여하는 이유는 AI에게 지금 요청한 작업의 목적이 무엇인지를 분명히 각인 시키기 위함입니다.
        - 여러 작업을 동시에 시키면 작업의 목적과 우선순위가 흐려지고, AI는 trade-off가 필요한 상황에서도 기준 없이 임의로 선택할 가능성이 높아집니다.
        - `어떻게 제어할까?`
            - 현재 작업의 목적을 정확히 명시 → 스프린트 계약 도입
            - 목적이 다른 작업은 같은 컨텍스트 안에 섞지 않고, 역할 또는 세션을 분리하여 처리 → 서브 에이전트 도입
2. `중간 컨텍스트 망각`
    - 프로젝트 분량이 커졌을 때 발생하는 문제
        - 초기 계획을 세우고 프로젝트를 진행하던 도중, 변경 사항이나 추가 사항이 발생해서 이를 반영하고 이후 작업을 진행하면 초기 의도, 중간 결정, 변경된 제약 조건이 현재 작업 지시에서 희석됩니다.
        - 이로 인해 방향성을 잃어버리고 이전 상황들을 고려하지 못한 채, 현재 지시한 작업을 수행할 가능성이 높습니다.
    - `어떻게 제어할까?`
        - 주요 결정 사항과 변경 사항을 워킹 메모리, 스프린트 문서 형태로 기록하여 컨텍스트를 보조
3. `자기 객관화 실패`
    - 앞서 언급한 주의력 분산과 컨텍스트 망각 단점과 연관되어있는 단점이기도 합니다.
        - 컨텍스트가 오염되거나 우선순위가 흐려지면, AI는 잘못된 전제를 바탕으로 그럴듯한 결과물을 생성할 수 있습니다.
    - `어떻게 제어할까?`
        - 핵심 로직은 테스트 케이스와 완료 기준을 먼저 작성한 뒤, Generator는 구현을 맡고 Reviewer는 테스트 통과 여부와 예외 케이스를 검증하도록 역할을 분리

## 3. 통제 전략

### 1. 서브 에이전트 분리

해야 할 작업 별로 에이전트를 분리하여 주의력이 분산 되는 것을 막습니다.

- MainAgent: 개발자와 직접 대화하며 계획 수립/피드백, Generator/Reviewer 호출 및 작업 위임, 워킹 메모리 문서 작성 등의 작업을 수행합니다.
- Generator: MainAgent가 스프린트 문서를 생성하여 Generator에게 전달하면 이를 기반으로 코드를 생성하는 작업을 수행하고, 스프린트 문서에 작업한 내용을 기록합니다
- Reviewer: 스프린트 문서를 확인하고, Generator가 생성한 코드를 검증하고 결과를 MainAgent에게 알립니다.

여기서 말하는 서브 에이전트는 반드시 물리적으로 분리된 AI 인스턴스를 의미하지는 않습니다. 도구가 지원한다면 실제 서브 에이전트로 분리할 수 있고, 그렇지 않다면 프롬프트·채팅 세션·작업 문서 단위로 역할을 분리해도 됩니다. 중요한 것은 “계획하는 역할”, “생성하는 역할”, “검증하는 역할”을 같은 컨텍스트 안에서 뒤섞지 않는 것입니다.

### 2. 스프린트 계약

작업을 작은 단위로 쪼개고 이걸 스프린트라고 지칭합니다.

스프린트 문서에서는 이번 작업에서 어떤 작업을 수행할 것이며 지켜야 할 룰과 완료 기준은 무엇 인지를 담습니다. 서브 에이전트(Generator)가 코드 생성을 마친 뒤에는 어떤 작업을 했는지를 스프린트 문서에 명시하여 메인 에이전트와 개발자가 작업 내용을 확인할 수 있도록 합니다.

`스프린트 문서 예시`

```
# Sprint 3: 로그인 실패 메시지 개선

## 목표
로그인 실패 시 사용자에게 원인을 구체적으로 안내한다.

## 제약조건
기존 로그인 API 응답 형식은 변경하지 않는다.

## 범위
- 이메일 미입력
- 비밀번호 미입력
- 인증 실패
- 서버 오류

## 제외 범위
- 소셜 로그인
- 비밀번호 재설정 플로우

## 완료 기준
- 각 실패 케이스별 메시지가 다르게 표시된다.
- 기존 로그인 성공 플로우는 변경하지 않는다.
- 관련 테스트가 통과한다.

## 작업 로그
- 변경한 파일
- 구현한 내용
- 확인한 테스트
- 남은 리스크
```

이를 통해 작업의 목적을 명확히 AI에게 인지 시킬 수 있고, 완성 기준이 정해져 있기 때문에 AI가 모호한 작업을 하는 것을 막을 수 있습니다.

## 4. 토큰 효율화/컨텍스트 관리 전략

### 1. 크리티컬 지점 분리

모든 작업에 엄격한 TDD와 테스트/검증 파이프라인을 적용하긴 어렵습니다. 토큰 소모량과 대기 시간을 감당할 수 없기 때문이죠. 그렇기 때문에 이를 적용할 부분과 적용하지 않을 부분을 분리합니다.

트랜잭션, 정합성이 중요한 데이터 수정 영역은 사람이 개입하여 엄격한 통제 속에 두고, 단순 UI/UX나 조회(READ)성 기능은 제약을 느슨하게 풀어 생산성을 높입니다.

이 지점들을 분리하는 과정은 메인 에이전트와 함께 계획을 세우는 과정에서 다양한 선택을 두고 trade-off를 고민하며 근거가 담겨야 합니다.

### 2. 컨텍스트 격리

스프린트가 거듭될수록 개발 로그 문서(`Sprint1.md`, `Sprint2.md`...)는 계속해서 누적되며 프로젝트의 자산이 됩니다. 하지만 그렇다고 서브 에이전트들에게 이 모든 히스토리 파일을 다 읽게 하면 토큰 비용이 기하급수적으로 늘어납니다.

메인 에이전트는 서브 에이전트에게 작업을 지시할 때, 과거의 모든 스프린트 파일을 넘기는 것이 아니라 기본적으로 현재 SprintN 문서를 우선 참조하고, 필요 시 과거 문서를 선별 참조하도록 컨텍스트를 제한합니다. 과거의 기록은 디렉토리에 고스란히 안전하게 보존하되, 에이전트가 바라보는 대화창(Context Window)의 부하는 격리하여 불필요한 토큰 낭비를 차단합니다.

### 3. 반복 작업은 스크립트 사용

코드로 분리할 수 있는 자동화 작업은 스크립트 파일을 만들어두고 단순히 이 파일을 실행 하는 것으로 단순화합니다.

예를들면 만들어진 테스트 코드를 실행하는 명령, 코드 컨벤션 검사하는 명령 등이 있겠습니다.

이를통해 AI에게 매번 긴 프롬프트로 린트나 빌드 규칙을 설명하는 대신, `npm test`나 `lint.sh` 같은 한 줄짜리 스크립트를 실행하게 만들어 컨텍스트 주입량을 줄일 수 있습니다.

## 5. 결론

사실 아직 “AI를 어떻게 활용하는 것이 가장 좋은 방법인가?”에 정답은 없다고 생각합니다.

저 역시 여러 자료들을 찾아보고, 직접 사용하면서 느꼈던 문제들을 바탕으로 지금의 방식들을 정리해본 것입니다.

AI는 분명 엄청난 생산성을 보여주지만, 동시에 목적이 흐려지거나 컨텍스트가 오염되면 예상과 다른 결과를 만들어내기도 합니다. 그래서 저는 단순히 “AI에게 일을 맡긴다”기보다는, 작업 단위를 나누고 역할을 분리하며 검증 구조를 만드는 방향으로 사용하고 있습니다.

물론 이 방식이 모든 프로젝트와 모든 개발자에게 정답이라고 생각하진 않습니다.

상황에 따라 더 단순한 방식이 맞을 수도 있고, 반대로 더 엄격한 통제 구조가 필요할 수도 있습니다.

다만 이 글에서 정리한 내용들이 AI를 개발에 활용하는 과정에서 시행착오를 줄이는 데 조금이라도 도움이 되었으면 좋겠습니다.
