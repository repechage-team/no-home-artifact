# NoHome 발표 대본 (Speaker Script)

> 슬라이드 번호는 `outline.md`의 9개 섹션 구조와 1:1 대응.
> 괄호 `( )`는 행동 지시 / 강조. `<!-- TODO -->`는 팀이 채울 자리.
> 말하기 속도 분당 ~300자 기준.

## 섹션 1. 기획 배경 · 목표

### [1-0] 타이틀
안녕하세요, 저희 팀이 만든 공공데이터 기반 아파트 실거래가 검색 서비스 **NoHome**을 소개하겠습니다.
저희가 풀고 싶었던 문제는 한 문장입니다 — (강조) **"실거래가 정보는 공개돼 있는데, 왜 일반인은 쓰기 어려울까?"**

### [1-1] 문제 정의
부동산 실거래가는 국토교통부가 공공데이터 API로 공개합니다. 그런데 (XML 가리키며) 실제 응답은 XML이고, 지역은 법정동코드 다섯 자리, 기간은 계약년월 여섯 자리로 넣어야 합니다.
게다가 매매, 전세, 월세가 서로 다른 API에 흩어져 있습니다.
"우리 동네 30평대 아파트 요즘 얼마지?"라는 단순한 질문에, 일반인은 사실상 접근하기 어렵습니다. 저희는 이 진입장벽을 없애는 데 집중했습니다.

### [1-2] 필수 기능 & 목표
저희 기준선은 과제 필수 요건입니다. 실거래가 수집과 검색, 회원 가입부터 조회·수정·삭제·로그인, 그리고 주택 정보 관리까지입니다.
데이터 전략은 (강조) "DB에 먼저 적재하고, 검색은 DB를 우선 조회하는" 하이브리드입니다. 회원 삭제는 과제 정책대로 물리 삭제로 구현했습니다.

### [1-3] 추가 기능 = AI 어시스턴트
그리고 저희는 필수를 넘어, 핵심 차별 기능을 하나 추가했습니다 — (강조) **AI 어시스턴트**입니다.
도입 동기는 명확합니다. 검색 UI를 잘 만들어도 결국 사용자는 필터를 일일이 조작해야 합니다. 저희는 이걸 자연어로 완전히 없애고 싶었습니다.
두 가지 동작을 합니다. 데이터를 묻는 질문, 그리고 화면을 직접 조작하는 명령입니다. 기술 기반은 Spring AI와 gpt-4o-mini이고, 질문모드에서 출발해 에이전트 MVP, 액션·필터 확장을 거쳐, 최종적으로 하나의 엔드포인트로 합치는 혼합형 재설계와 단기 대화기억까지 구현했습니다.

## 섹션 4. 개발 결과

### [4-1] 필수 기능 구현 현황
먼저 과제 필수 요건입니다. (표 가리키며)
실거래가 수집과 검색, 회원 CRUD와 로그인, 주택 정보 관리, 그리고 지도 표시까지 필수 기능을 모두 구현하고 검증했습니다. 빠짐없이 채웠다는 걸 먼저 말씀드립니다.

### [4-2] 핵심기능 ① 실거래가 검색 + 자동 임포트
검색은 매매, 전세, 월세, 전월세, 전체 다섯 가지 거래유형을 지원합니다.
지역은 시도에서 시군구, 법정동까지 3단계로 좁히고, 가격은 이중 슬라이더로 조절합니다. 거래유형에 따라 노출되는 가격 필터가 달라지도록 설계했습니다 — 전월세를 위해 보증금과 월세 컬럼까지 스키마를 확장했습니다.
검색은 DB를 우선 조회하고, 부족한 구간만 공공데이터에서 자동으로 채웁니다. 이 알고리즘은 뒤 섹션에서 자세히 보겠습니다.

### [4-3] 핵심기능 ② 회원 & 인증
회원은 가입, 조회, 수정, 물리 삭제, 로그인, 로그아웃을 모두 구현했습니다.
보안 측면에서, 비밀번호는 BCrypt로 해싱하고, 인증은 JWT access·refresh 토큰을 HttpOnly·Secure 쿠키로 운반합니다. 로그아웃 시 refresh 토큰을 블랙리스트에 등록하고, 운영 환경에서는 보안 설정이 빠지면 기동을 막는 fail-closed 검증을 넣었습니다.
또 회원 검색 API는 관리자 권한으로 분리해, 일반 회원이 호출하면 403을 반환하도록 했습니다.

### [4-4] 추가기능 ③ AI 어시스턴트 — 구현
AI 어시스턴트는 (강조) **단 하나의 엔드포인트 `/api/ai/assistant`**로 동작합니다. 예전엔 질문용과 조작용 엔드포인트가 나뉘고 모드 토글이 있었는데, 이걸 없애고 LLM이 tool calling으로 직접 분기하도록 재설계했습니다.
그래서 "강남구 평균가 얼마야?"엔 텍스트로 답하고, "전세로 바꾸고 보증금 3억 이하만"엔 직접 필터를 바꾸고 검색을 실행합니다. 사용자는 모드를 신경 쓸 필요가 없습니다.
액션은 검색, 필터설정, 페이지 이동, 항목 선택, 지도 포커스, 초기화 여섯 가지에 되묻기 가드를 더했고, 필터는 아홉 가지를 지원합니다.

### [4-5] 검증 현황
품질은 테스트로 증명했습니다.
최신 기준으로 (강조) **백엔드 177건, 프론트엔드 51건, 합쳐서 228건의 테스트가 모두 통과**했습니다. E2E와 브라우저 시나리오도 검증했습니다.
(추이 차트 가리키며) 기능을 확장하면서 테스트도 함께 늘려, 회귀를 막으며 안전하게 재설계할 수 있었습니다.

## 섹션 5. 개발 환경 & 시스템 구조도

### [5-1] 전체 시스템 구조도
구조는 표준 3-tier입니다. Vue3 SPA가 REST API를 호출하고, Spring Boot가 MyBatis로 MySQL을 조회합니다. 개발 단계에서는 Vite 프록시가 `/api`를 백엔드로 넘깁니다.
외부 연동은 세 가지입니다 — 국토부 공공데이터 API, LLM용 SSAFY GMS, 지도용 Kakao Map입니다.
전체는 Docker Compose 하나로 백엔드, 프론트엔드, MySQL을 함께 띄울 수 있습니다.

### [5-2] 개발 환경 & 기술 스택
스택은 과제 요구인 Spring Boot, MyBatis, Vue를 충실히 따랐습니다. 백엔드는 Java 17, Spring Boot 3.5, 프론트는 Vue 3.5와 Vite입니다.
한 가지 더, AI 파트는 최신 Spring AI를 도입했습니다. 협업은 Git 브랜치 전략과 문서 기반 Sprint로 운영했고, 비밀 키는 모두 환경변수로 분리해 저장소에 커밋하지 않았습니다.

## 섹션 7. 적용 패턴 및 핵심 알고리즘

### [7-1] Tool Calling + returnDirect 패턴
AI의 핵심은 Spring AI의 **Tool Calling**입니다. 하나의 엔드포인트에서 LLM에게 두 종류의 도구를 쥐여줬습니다.
데이터 조회 도구는 결과를 LLM이 텍스트로 요약해 답변이 되고, 페이지 액션 도구 여섯 개는 (강조) `returnDirect`로 실행 명령을 구조화된 JSON으로 곧장 돌려줍니다. 응답의 finishReason이 "returnDirect"인지로 명령인지 답변인지를 결정적으로 구분합니다.
중요한 설계는, "아무 도구도 부르지 않는 것"도 1급 선택지로 둔 점입니다. 덕분에 불만이나 모호한 말을 억지로 액션으로 오분류하는 문제를 구조적으로 줄였습니다. 이 거동은 Spring AI 1.1.2에서 실제로 동작하는지 PoC로 먼저 실측한 뒤 정식 구현했습니다.

### [7-2] Capability-driven Agent
두 번째 패턴은 capability 기반 에이전트입니다. 원칙은 (강조) "단일 출처 + 프론트 최종 강제"입니다.
프론트가 현재 화면 상태와 지원 가능한 동작 목록을 함께 보내면, LLM은 그걸 힌트로만 쓰고, 실제 적용은 프론트가 합니다. 인식하는 필터만 적용하고, 모르는 건 버린 뒤 사용자에게 안내합니다.
효과가 큽니다. 전월세 같은 새 필터를 추가해도 선언 한 곳만 고치면 에이전트가 자동으로 적응하고, 폼과 선언이 어긋나지 않게 동기화 테스트로 강제했습니다. 한 가지 교훈은, 거래월 범위 판단을 LLM에 맡겼더니 정상 월도 거부하는 환각이 생겨서, 그건 서버의 결정적 가드로 옮겼다는 점입니다.

### [7-3] 단기 대화기억 & 저장소 선택 근거
세 번째는 멀티턴을 가능하게 하는 단기 대화기억입니다. "마포구 시세 알려줘" 다음에 "방금 그 지역 전세는?"이라고 해도 맥락을 이어갑니다.
구현은 Spring AI의 메모리에 최근 10개 메시지를 담는 방식이고, 대화 키는 회원 ID와 세션 UUID를 합쳐 사용자와 세션을 격리합니다. 대화 원문은 (강조) 서버 메모리에만 있고 DB에 저장하지 않습니다.
여기서 저희가 고민한 게 저장소 선택입니다. InMemory, 브라우저 localStorage, RDB, Redis 네 가지를 비교했는데, "창을 닫으면 초기화되고 원문을 영속하지 않는다"는 요구에는 InMemory가 가장 잘 맞았습니다. 의존성이 없고 가장 빠르며 프라이버시 부담이 없으니까요. 다만 스케일아웃하면 Redis와 TTL로, 영구 보존이 필요해지면 RDB로 — 빈 한 곳만 바꾸면 전환되도록 추상화해 뒀습니다.

### [7-4] DB 우선 + 라이브 검색 성능 개선
네 번째는 데이터 전략입니다. 기본은 "DB 우선, 부족분만 자동 적재"이고, `api_row_hash`로 중복을 걸러 멱등성을 보장합니다.
그런데 캐시가 없는 지역을 처음 조회하면 17초까지 걸리는 문제가 있었습니다. 강남구 한 달 전월세가 1,500건이 넘는데, 그걸 다 저장하고 나서야 응답했기 때문입니다.
그래서 응답 경로와 저장 경로를 분리했습니다. (강조) 공공 API 결과를 서버 메모리에서 바로 가공해 먼저 응답하고, DB 저장은 백그라운드에서 배치로 처리합니다. 저장도 한 건씩 왕복하던 걸 배치 upsert로 바꾸고, API 페이지 크기를 100에서 1000으로 키워 호출 수를 줄였습니다. 결과적으로 첫 조회 지연이 크게 완화됐습니다.

### [7-5] 보안·인코딩 패턴
마지막으로 기반 패턴들입니다. 인증은 JWT와 BCrypt에 운영 fail-closed 검증을 더했고, 회원 검색은 관리자 권한으로 분리했습니다. 지역은 서울 25개 자치구 resolver와 법정동 카탈로그 머지로 매핑합니다.
인코딩은 공공데이터 한글이 깨지는 mojibake 문제를 DB 읽기 시 보정합니다 — 이건 실제로 AI 환각처럼 보였던 문제와도 연결됩니다. 뒤에서 회고로 다루겠습니다.

## 섹션 8. 기대 효과

### [8-1] 기대 효과
정리하면 기대 효과는 네 가지입니다.
**사용자 측면**에서는 공공데이터 진입장벽을 없애 누구나 실거래가를 탐색하고, 자연어로 손쉽게 조작합니다.
**기술 측면**에서는 Spring AI Tool Calling을 실전에 적용했고, DB 우선과 자동 적재로 응답성과 정합성을 함께 잡았습니다.
**확장성 측면**에서는 capability 기반 구조라 신규 기능을 저비용으로 붙일 수 있고, **운영 측면**에서는 개인정보 미기록 정책과 fail-closed 보안으로 안전하게 배포할 수 있습니다.

## 섹션 9. 개발 후기

### [9-1] 트러블슈팅 회고
실제로 부딪힌 문제 몇 가지를 공유하겠습니다.
첫째, 공공데이터 적재 시 142건 중 10건만 들어왔습니다. 페이지네이션 누락이 원인이라 전체 페이지를 순회하도록 고쳤습니다.
둘째, 강남구 데이터가 있는데도 "조회할 수 없다"고 나왔습니다. 정상 응답코드 "000"을 실패로 오분류한 거라, 명세를 실측해 화이트리스트로 바로잡았습니다.
셋째, 캐시 없는 지역 첫 조회가 17초나 걸렸습니다. 저장을 다 마치고 응답했기 때문이라, 응답을 먼저 보내고 저장은 백그라운드 배치로 돌려 크게 개선했습니다.
넷째, 검색하면 지도가 사라졌습니다. Vue 리렌더가 Kakao의 비-Vue DOM을 지운 거라, 지도 상태를 reactive 바깥으로 뺐습니다.
다섯째, AI가 아파트명만 매번 다르게 답하는 환각처럼 보였는데, 사실은 seed 데이터 더블 인코딩이 원인이었습니다. 인코딩을 고치니 정확도가 복구됐습니다.

## 예상 질문 & 답변 (Q&A 대비)

**Q. 로그인이 세션 기반이라면서 JWT를 쓰는 이유는?**
A. 인증 컨셉은 로그인 기반이지만, 구현은 무상태 JWT를 HttpOnly 쿠키로 운반합니다. 서버 세션 저장 부담 없이 토큰 만료·refresh·블랙리스트로 로그인 수명을 제어합니다.

**Q. AI가 잘못된 명령을 내리면?**
A. 프론트가 응답 필터를 검증해 지원하는 값만 적용하고(capability-driven), 서울·거래월 가드와 입력 제한을 둡니다. 실제 실행 주체가 프론트라 LLM이 임의로 시스템을 바꾸지 못합니다.

**Q. GMS 키가 없으면 앱 전체가 죽지 않나요?**
A. 초기에 그 문제가 있었습니다. 지금은 키가 비면 AI 채팅만 503으로 graceful 처리하고, 부동산 검색 등 나머지 기능은 정상 기동하도록 EnvironmentPostProcessor로 분리했습니다.

**Q. 공공데이터 호출이 느리거나 실패하면?**
A. 검색은 DB 우선이라 적재된 구간은 즉시 응답합니다. 자동 적재 실패는 KEY_MISSING·QUOTA·TIMEOUT 등으로 분류해 안내하고, 응답코드 03(데이터 없음)은 정상 빈 결과로 처리합니다.

**Q. 데이터 중복은 어떻게 막나요?**
A. 원본 row 주요 필드를 조합한 `api_row_hash`에 유니크 제약을 걸어 skip하고, import batch 이력으로 완료 구간 재호출도 차단합니다(멱등).

**Q. AI 대화에 개인정보가 저장되나요?**
A. 아니요. 단기 대화기억은 서버 메모리(InMemory)에만 최근 10개를 두고 DB에 영속하지 않습니다. 창을 닫으면 세션 UUID가 사라져 초기화되고, 서버 재기동 시에도 휘발됩니다. 로깅도 질문·답변 원문과 조회 결과는 기록하지 않고 건수·토큰 수 같은 메타데이터만 남깁니다.

**Q. 왜 대화기억을 Redis나 DB가 아니라 InMemory로 했나요?**
A. 네 가지(InMemory/localStorage/RDB/Redis)를 비교했는데, "닫으면 초기화 + 원문 비영속"이라는 요구에는 InMemory가 최적이었습니다. 단일 백엔드 인스턴스라 추가 인프라가 불필요하고 프라이버시 부담도 가장 낮습니다. 스케일아웃하면 Redis+TTL, 영구 보존이 필요해지면 RDB로, 빈 한 곳만 바꿔 전환할 수 있게 추상화했습니다.

**Q. AI가 "3페이지 가서 2번째 선택"처럼 여러 동작을 한 번에 시키면?**
A. 현재 returnDirect는 호출당 한 액션이라 첫 동작만 실행됩니다. 결과 기반 선택(예: 면적 가장 작은 매물)과 다중 동작은 결과 요약 동봉·수동 tool 루프로 푸는 후속 과제로 백로그에 있습니다. 솔직한 현재 한계로 말씀드립니다.
