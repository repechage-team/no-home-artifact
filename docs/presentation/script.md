# NoHome 발표 대본 (Speaker Script)

> 슬라이드 번호는 `outline.md`의 9개 섹션 구조와 1:1 대응.
> 괄호 `( )`는 행동 지시 / 강조. `<!-- TODO -->`는 팀이 채울 자리.
> 말하기 속도 분당 ~300자 기준.

## 섹션 1. 기획 배경 · 목표

### [1-0] 타이틀
안녕하세요, 저희 팀이 만든 공공데이터 기반 아파트 실거래가 검색 서비스 **NoHome**을 소개하겠습니다.
저희가 풀고 싶었던 문제는 한 문장입니다 — (강조) **"실거래가 정보는 공개돼 있는데, 왜 일반인은 쓰기 어려울까?"**

### [1-1] 문제 정의
부동산 실거래가는 국토교통부가 공공데이터 API로 공개합니다. 그런데 (XML 가리키며) 실제 응답은 XML이고, 지역은 법정동코드 다섯 자리, 기간은 계약년월 여섯 자리로 넣어야 합니다.
게다가 매매, 전세, 월세가 서로 다른 API에 흩어져 있습니다.
"우리 동네 30평대 아파트 요즘 얼마지?"라는 단순한 질문에, 일반인은 사실상 접근하기 어렵습니다. 저희는 이 진입장벽을 없애는 데 집중했습니다.

### [1-2] 필수 기능 & 목표
저희 기준선은 과제 필수 요건입니다. 실거래가 수집과 검색, 회원 가입부터 조회·수정·삭제·로그인, 그리고 주택 정보 관리까지입니다.
데이터 전략은 (강조) "DB에 먼저 적재하고, 검색은 DB를 우선 조회하는" 하이브리드입니다. 회원 삭제는 과제 정책대로 물리 삭제로 구현했습니다.

### [1-3] 추가 기능 = AI 어시스턴트
그리고 저희는 필수를 넘어, 핵심 차별 기능을 하나 추가했습니다 — (강조) **AI 어시스턴트**입니다.
도입 동기는 명확합니다. 검색 UI를 잘 만들어도 결국 사용자는 필터를 일일이 조작해야 합니다. 저희는 이걸 자연어로 완전히 없애고 싶었습니다.
두 가지 모드로 만들었습니다. 데이터를 묻는 질문 모드, 그리고 화면을 직접 조작하는 에이전트 모드입니다. 기술 기반은 Spring AI와 gpt-4o-mini이고, 사흘에 걸쳐 질문모드 → 에이전트 MVP → 액션·필터 확장 순으로 단계적으로 구현했습니다.

## 섹션 4. 개발 결과

### [4-1] 필수 기능 구현 현황
먼저 과제 필수 요건입니다. (표 가리키며)
실거래가 수집과 검색, 회원 CRUD와 로그인, 주택 정보 관리, 그리고 지도 표시까지 필수 기능을 모두 구현하고 검증했습니다. 빠짐없이 채웠다는 걸 먼저 말씀드립니다.

### [4-2] 핵심기능 ① 실거래가 검색 + 자동 임포트
검색은 매매, 전세, 월세, 전월세, 전체 다섯 가지 거래유형을 지원합니다.
지역은 시도에서 시군구, 법정동까지 3단계로 좁히고, 가격은 이중 슬라이더로 조절합니다. 거래유형에 따라 노출되는 가격 필터가 달라지도록 설계했습니다 — 전월세를 위해 보증금과 월세 컬럼까지 스키마를 확장했습니다.
검색은 DB를 우선 조회하고, 부족한 구간만 공공데이터에서 자동으로 채웁니다. 이 알고리즘은 뒤 섹션에서 자세히 보겠습니다.

### [4-3] 핵심기능 ② 회원 & 인증
회원은 가입, 조회, 수정, 물리 삭제, 로그인, 로그아웃을 모두 구현했습니다.
보안 측면에서, 비밀번호는 BCrypt로 해싱하고, 인증은 JWT access·refresh 토큰을 HttpOnly·Secure 쿠키로 운반합니다. 로그아웃 시 refresh 토큰을 블랙리스트에 등록하고, 운영 환경에서는 보안 설정이 빠지면 기동을 막는 fail-closed 검증을 넣었습니다.

### [4-4] 추가기능 ③ AI 어시스턴트 — 구현
AI 어시스턴트는 두 모드로 동작합니다. (비교표 가리키며)
질문 모드는 "강남구 평균가 얼마야?"에 텍스트로 답하고, 에이전트 모드는 "전세로 바꾸고 보증금 3억 이하만"에 직접 필터를 바꾸고 검색을 실행합니다.
액션은 검색, 필터설정, 페이지 이동, 항목 선택, 지도 포커스, 초기화, 되묻기까지 일곱 가지, 필터는 아홉 가지를 지원합니다. MVP에서 시작해 Phase 2에서 액션과 필터를 확장했습니다.

### [4-5] 검증 현황
품질은 테스트로 증명했습니다.
백엔드 116건, 프론트엔드 44건, 합쳐서 (강조) **160건의 테스트가 모두 통과**했고, E2E 9건도 전부 통과했습니다.
사용자 시나리오 14개를 풀스택 환경에서 실데이터로 검증했습니다. (추이 차트 가리키며) 기능을 확장하면서 테스트를 78건에서 116건까지 함께 늘렸습니다.

## 섹션 5. 개발 환경 & 시스템 구조도

### [5-1] 전체 시스템 구조도
구조는 표준 3-tier입니다. Vue3 SPA가 REST API를 호출하고, Spring Boot가 MyBatis로 MySQL을 조회합니다. 개발 단계에서는 Vite 프록시가 `/api`를 백엔드로 넘깁니다.
외부 연동은 세 가지입니다 — 국토부 공공데이터 API, LLM용 SSAFY GMS, 지도용 Kakao Map입니다.
전체는 Docker Compose 하나로 백엔드, 프론트엔드, MySQL을 함께 띄울 수 있습니다.

### [5-2] 개발 환경 & 기술 스택
스택은 과제 요구인 Spring Boot, MyBatis, Vue를 충실히 따랐습니다. 백엔드는 Java 17, Spring Boot 3.5, 프론트는 Vue 3.5와 Vite입니다.
한 가지 더, AI 파트는 최신 Spring AI를 도입했습니다. 협업은 Git 브랜치 전략과 문서 기반 Sprint로 운영했고, 비밀 키는 모두 환경변수로 분리해 저장소에 커밋하지 않았습니다.

## 섹션 7. 적용 패턴 및 핵심 알고리즘

### [7-1] Tool Calling + returnDirect 패턴
AI의 핵심은 Spring AI의 **Tool Calling**입니다. LLM에게 두 종류의 도구를 쥐여줬습니다.
데이터 조회 도구는 결과를 LLM이 텍스트로 요약해 답변이 되고, 페이지 액션 도구 여섯 개는 (강조) `returnDirect`로 실행 명령을 구조화된 JSON으로 곧장 돌려줍니다.
중요한 설계는, "아무 도구도 부르지 않는 것"도 1급 선택지로 둔 점입니다. 덕분에 단순 대화를 억지로 액션으로 오분류하는 문제를 구조적으로 줄였습니다.

### [7-2] Capability-driven Agent
두 번째 패턴은 capability 기반 에이전트입니다. 원칙은 (강조) "단일 출처 + 프론트 최종 강제"입니다.
프론트가 현재 화면 상태와 지원 가능한 동작 목록을 함께 보내면, LLM은 그걸 힌트로만 쓰고, 실제 적용은 프론트가 합니다. 인식하는 필터만 적용하고, 모르는 건 버린 뒤 사용자에게 안내합니다.
효과가 큽니다. 메인 화면에 새 필터를 추가해도 선언 한 곳만 고치면 에이전트가 자동으로 적응합니다. 안전장치로 대화 기억은 휘발성이라 개인정보를 저장하지 않고, 분당 10회·500자 제한과 서울·거래월 가드를 뒀습니다.

### [7-3] 자동 임포트 & 멱등 적재
데이터 전략의 핵심은 (강조) "DB 우선, 부족분만 자동 적재"입니다.
검색이 들어오면 DB 커버리지를 판단하고, 없으면 공공데이터 API를 호출해 XML을 파싱하고, `api_row_hash`로 중복을 걸러 적재한 뒤 다시 조회합니다.
이미 적재한 구간은 import 이력으로 건너뛰어 멱등성을 보장합니다. 전월세는 월세가 0원이면 전세, 넘으면 월세로 매핑합니다. 참고로 공공데이터 정상 코드가 "00"과 "000" 두 가지라, 실측 근거로 둘 다 성공으로 처리하도록 고쳤습니다.

### [7-4] 보안·인코딩 패턴
마지막으로 기반 패턴들입니다. 인증은 JWT와 BCrypt에 운영 fail-closed 검증을 더했고, 지역은 서울 25개 자치구 resolver와 법정동 카탈로그 머지로 매핑합니다.
인코딩은 공공데이터 한글이 깨지는 mojibake 문제를 DB 읽기 시 보정하도록 처리했습니다 — 이건 실제로 AI 환각처럼 보였던 문제와도 연결됩니다. 뒤에서 회고로 다루겠습니다.

## 섹션 8. 기대 효과

### [8-1] 기대 효과
정리하면 기대 효과는 네 가지입니다.
**사용자 측면**에서는 공공데이터 진입장벽을 없애 누구나 실거래가를 탐색하고, 자연어로 손쉽게 조작합니다.
**기술 측면**에서는 Spring AI Tool Calling을 실전에 적용했고, DB 우선과 자동 적재로 응답성과 정합성을 함께 잡았습니다.
**확장성 측면**에서는 capability 기반 구조라 신규 기능을 저비용으로 붙일 수 있고, **운영 측면**에서는 개인정보 미기록 정책과 fail-closed 보안으로 안전하게 배포할 수 있습니다.

## 섹션 9. 개발 후기

### [9-1] 트러블슈팅 회고
실제로 부딪힌 문제 몇 가지를 공유하겠습니다.
첫째, 공공데이터 적재 시 142건 중 10건만 들어왔습니다. 페이지네이션 누락이 원인이라 전체 페이지를 순회하도록 고쳤습니다.
둘째, 강남구 데이터가 있는데도 "조회할 수 없다"고 나왔습니다. 정상 응답코드 "000"을 실패로 오분류한 거라, 명세를 실측해 화이트리스트로 바로잡았습니다.

정리하겠습니다. NoHome은 (강조) **"공공데이터 실거래가를, API를 몰라도, 말 한마디로 검색하는"** 서비스입니다.
필수 요건을 모두 충족하면서, 자동 임포트와 AI 어시스턴트로 차별화했습니다. 감사합니다. 질문 받겠습니다.

## 예상 질문 & 답변 (Q&A 대비)

**Q. 로그인이 세션 기반이라면서 JWT를 쓰는 이유는?**
A. 인증 컨셉은 로그인 기반이지만, 구현은 무상태 JWT를 HttpOnly 쿠키로 운반합니다. 서버 세션 저장 부담 없이 토큰 만료·refresh·블랙리스트로 로그인 수명을 제어합니다.

**Q. AI가 잘못된 명령을 내리면?**
A. 프론트가 응답 필터를 검증해 지원하는 값만 적용하고(capability-driven), 서울·거래월 가드와 입력 제한을 둡니다. 실제 실행 주체가 프론트라 LLM이 임의로 시스템을 바꾸지 못합니다.

**Q. GMS 키가 없으면 앱 전체가 죽지 않나요?**
A. 초기에 그 문제가 있었습니다. 지금은 키가 비면 AI 채팅만 503으로 graceful 처리하고, 부동산 검색 등 나머지 기능은 정상 기동하도록 EnvironmentPostProcessor로 분리했습니다.

**Q. 공공데이터 호출이 느리거나 실패하면?**
A. 검색은 DB 우선이라 적재된 구간은 즉시 응답합니다. 자동 적재 실패는 KEY_MISSING·QUOTA·TIMEOUT 등으로 분류해 안내하고, 응답코드 03(데이터 없음)은 정상 빈 결과로 처리합니다.

**Q. 데이터 중복은 어떻게 막나요?**
A. 원본 row 주요 필드를 조합한 `api_row_hash`에 유니크 제약을 걸어 skip하고, import batch 이력으로 완료 구간 재호출도 차단합니다(멱등).

**Q. AI 대화에 개인정보가 저장되나요?**
A. 아니요. 대화 기억은 휘발성이고, 로깅 정책상 질문·답변 원문과 조회 결과(아파트명·가격 등)는 기록하지 않습니다. 진단 시에도 건수·토큰 수 같은 메타데이터만 남깁니다.
