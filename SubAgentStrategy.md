AI 기반 개발(Vibe Coding) 환경에서 멀티 에이전트 크루가 단기 기억을 잃지 않고, 토큰을 효율적으로 소비하며, 개발자와 유기적으로 소통할 수 있도록 설계된 통합 워크플로우 가이드라인입니다.

## 1. 전역 아키텍처: 문서화 파이프라인 (Separation of Concerns)

AI의 주의력 분산과 토큰 낭비를 막기 위해 역할(기획, 기술 규칙, 일정, 실시간 로그)에 따라 문서를 완전히 분리합니다.

### 프로젝트 디렉토리 구조

```
my-project/
├── CLAUDE.md (또는 AGENTS.md)    # [하네스] 전역 행동 지침 및 라우팅 규칙
└── docs/
    ├── PRD.md                  # [기획] 사용자 요구사항 및 비즈니스 명세
    ├── spec.md                 # [스펙] 아키텍처, DB, API 등 기술 기준점
    ├── plan.md                 # [계획] 거시적 마일스톤 로드맵 및 결정 체크포인트
    ├── working-memory.md       # [지식 베이스] 실시간 작업 현황, 완료 Sprint 요약, 트러블슈팅 아카이브
    └── sprints/                # [작동 기억] Sprint별 구현 지시 계약 및 개발 로그
        ├── Sprint1.md          # 완료된 스프린트 히스토리
        └── Sprint2.md          # 현재 세션 활성 메모리
```

### 문서별 세부 역할 명세

### CLAUDE.md / AGENTS.md (최상위 하네스)

- **역할:** 에이전트 전역 행동 제어 및 프로세스 가드레일 (Behavioral Lint).
- **핵심 지침:** 코드 컨벤션, 깃 커밋 메시지 규칙, 도구 사용 통제. AI가 임의로 스프린트 파일을 생성하거나 파이프라인을 탈선하지 못하도록 묶어두는 프로젝트의 헌법 역할을 합니다.

### docs/PRD.md (Product Requirement Document)

- **역할:** 사용자 관점의 기획 및 요구사항 명세.
- **핵심 지침:** 기술적 용어를 배제하고 스토리텔링 줄글 중심으로 작성합니다. AI가 서비스의 궁극적인 목적을 이해하고, 구현 중 모호한 기획적 예외 상황이 터졌을 때 의도에 맞는 올바른 판단을 내리도록 돕습니다.

### docs/spec.md (Technical Specification)

- **역할:** 변하지 않는 기술적 뼈대와 규칙 정의.
- **핵심 지침:** 시스템 아키텍처 및 환경, DB 스키마, API 엔드포인트 규격, 패키지 구조 규칙 등을 명시합니다. AI가 코드를 짤 때 절대 어겨서는 안 되는 절대적인 기술 기준점입니다.

### docs/plan.md (Master Plan)

- **역할:** 여러 Sprint를 묶는 거시적 Milestone 로드맵과 주요 결정 체크포인트 관리.
- **핵심 지침:** Sprint 목록이나 세부 백로그를 적는 문서가 아닙니다. 큰 작업 단위인 Milestone의 목표, 산출물 후보, 완료 판단 기준, 선행 결정 항목만 유지하여 텍스트 비대화를 막습니다. Sprint 단위의 목표, 범위, 완료 기준은 `docs/sprints/SprintN.md`에 기록합니다.

### docs/working-memory.md (Working Memory / 지식 베이스)

- **역할:** 개발자 관점의 실시간 작업 현황 파악 및 누적 트러블슈팅 아카이브.
- **핵심 지침:** 메인 에이전트가 전담하여 관리하는 문서입니다. 최상단에는 현재 어떤 에이전트가 어떤 작업을 하고 있는지 거시적인 대시보드를 유지하고, 활성 Sprint와 완료된 Sprint들의 핵심 결론을 누적합니다. 하단에는 지나간 Sprint의 핵심 트러블슈팅 내역을 정리합니다. 개발자가 프로젝트 전체 통제권을 쥐기 위한 핵심 가이드 역할을 합니다.

### docs/sprints/SprintN.md (Sprint Contract / Sprint Log)

- **역할:** Manager가 확정한 Sprint 구현 지시 계약과 Generator/Reviewer의 실행 로그를 함께 보존한다.
- **핵심 지침:** Sprint 문서는 단순 결과 로그가 아니라 해당 Sprint에서 Manager가 무엇을 구현하라고 지시했는지 명시하는 기준 계약입니다. Manager는 Generator를 호출하기 전에 목표, 범위, 제외 범위, 완료 기준, 확정 기술 결정, 작업 지시, 검증 지시를 먼저 작성합니다. Generator는 이 문서를 읽고 지시된 범위만 구현하며, 변경 파일, 구현 내용, 발견된 이슈, 시도한 해결책, 에러, 남은 리스크를 같은 문서에 이어 기록합니다. Reviewer는 동일 문서를 기준으로 검증하고 Pass / Fail과 누락 테스트를 기록합니다. Sprint가 종료되면 해당 파일은 히스토리로 보존하고, 다음 Sprint는 새 번호 파일로 시작하여 AI의 단기 기억 유실을 막고 토큰을 획기적으로 아낍니다.

Sprint 문서의 권장 구성:

- Sprint 목표와 Milestone 연결.
- Manager 구현 지시.
- 범위와 제외 범위.
- 확정 기술 결정과 아직 결정하지 않을 항목.
- Generator 작업 지시.
- Reviewer 검증 지시.
- 완료 기준.
- 작업 로그.
- 변경 파일 목록.
- 검증 결과.
- 남은 리스크와 인계 사항.

운영 원칙:

- 프롬프트는 실행 지시이고, `SprintN.md`는 계약이다.
- Generator에게 전달하는 프롬프트는 새 요구사항을 길게 재정의하지 않고, `docs/spec.md`와 활성 `docs/sprints/SprintN.md`를 읽고 그 범위만 구현하라고 지시한다.
- Sprint 진행 중 범위 변경이 필요하면 먼저 Manager가 `SprintN.md`의 계약을 갱신한 뒤 Generator에게 재작업을 지시한다.

## 2. 조직 아키텍처: 멀티 에이전트 협업 전략 (Agentic Workflow)

문서가 완벽히 분리되었기 때문에, 각 에이전트에게 필요한 파일의 맥락만 주입(Context Distribution)하여 효율을 극대화하는 에이전틱 워크플로우를 구축합니다.

```
       [ 개발자 ]
           │
           ▼
┌──────────────────────────┐
│  1. 메인 에이전트 (Manager) │ ◄─── 지휘, 문서 제어 (PRD/spec/plan/working-memory)
└─────┬──────────────┬─────┘
      │              │
      │ 2. 업무 할당  │ 4. 검증 지시
      ▼              ▼
┌────────────┐ ┌────────────┐
│ Generator  │ │  Reviewer  │
│ (에이전트1) │ │ (에이전트2) │
└─────┬──────┘ └─────┬──────┘
      │ 3. 구현      │ 5. 테스트
      ▼              ▼
 [ 소스 코드 ]   [ 테스트 코드 ]
      │              │
      └──────►───────┴─────► SprintN.md에 실시간 로깅
```

### 에이전트별 페르소나 및 행동 지침

### 1. 메인 에이전트 (Manager Agent)

- **책임:** 개발자와 소통, 전체 마일스톤 관리, 서브 에이전트 오케스트레이션 및 `working-memory.md` 전담 관리.
- **행동 방식:** 직접 코드를 짜지 않아 대규모 컨텍스트를 항상 이성적인 상태로 유지합니다. 서브 에이전트들에게 업무를 배분할 때 필요한 파일만 읽도록 지정하고, 상태 변화에 따라 `working-memory.md` 최상단의 대시보드를 실시간으로 갱신합니다.

### 2. 서브 에이전트 1: 생성기 (Generator Agent)

- **책임:** `spec.md`와 `SprintN.md`를 기반으로 실제 기능 구현 및 소스 코드 작성.
- **행동 방식:** 오직 코드 생성에만 집중하며, 에러를 만나면 수정 과정과 시도한 방법을 `SprintN.md`에 개발자 수준의 날것 그대로 상세히 로깅합니다.

### 3. 서브 에이전트 2: 리뷰어 (Reviewer / Tester Agent)

- **책임:** 구현된 코드 검증, 테스트 코드 작성 및 실행.
- **행동 방식:** 구현과 검증의 분리 원칙에 따라, Generator가 짠 코드를 제3자의 눈으로 테스트합니다. `spec.md` 기술 규격과 `SprintN.md` 목표를 달성했는지 확인 후 Pass / Fail 결과를 메인 에이전트에게 보고합니다.

## 3. 엔드투엔드 워크플로우 (End-to-End Workflow)

개발자가 명령을 내린 시점부터 서브 에이전트들이 움직이고, 최종적으로 문서가 업데이트되는 전체 진행 흐름입니다.

### Phase 1: 준비 및 시작 (개발자 → 메인 에이전트)

1. **개발자:** 메인 에이전트에게 새로운 요구사항이나 스프린트 시작을 선언합니다.
    - 예: "메인아, 이제 게시글 기능 개발하는 Sprint 2 시작하자. PRD.md랑 spec.md 기반으로 세팅해줘."
2. **메인 에이전트:** `plan.md`에서 현재 Milestone과 결정 체크포인트를 확인하고, 이번에 활성화할 `docs/sprints/Sprint2.md` 파일을 준비합니다.
3. **Sprint 계약 작성:** 메인 에이전트는 Generator를 호출하기 전에 `Sprint2.md`에 이번 Sprint의 구현 지시를 명시합니다. 이 문서에는 목표, 범위, 제외 범위, 완료 기준, 확정 기술 결정, Generator 작업 지시, Reviewer 검증 지시, 작업 로그 섹션이 포함되어야 합니다.
4. **대시보드 초기화:** 메인 에이전트는 개발자가 보는 현황판인 `working-memory.md` 최상단의 실시간 작업 현황 대시보드를 즉시 업데이트합니다. (현재 활성화된 스프린트: Sprint 2 / 메인 에이전트: 가이드 수립 중 / 서브 에이전트들: 대기 상태로 표기)

### Phase 2: 개발 및 실시간 로깅 (메인 → Generator)

1. **업무 할당:** 메인 에이전트가 Generator를 깨우며 컨텍스트를 지정해 줍니다.
    - 지침: "`docs/spec.md`와 `docs/sprints/Sprint2.md`를 읽고, Sprint2 문서의 Manager 구현 지시와 Generator 작업 지시 범위만 구현해라. 변경 파일, 구현 내용, 에러, 남은 리스크는 Sprint2.md에 기록해라."
2. **대시보드 최신화:** 메인이 일을 시켰으므로, `working-memory.md` 대시보드를 [Generator: 작업 중 - 게시글 작성 API 구현]으로 상태를 변경하여 개발자가 실시간 현황을 볼 수 있게 합니다.
3. **코드 구현 및 미시 로깅:** Generator는 `Sprint2.md`의 계약을 기준으로 코드를 작성합니다. 중간에 발생하는 에러나 예기치 못한 이슈, 수정 과정은 `Sprint2.md`에 상세히 로깅합니다. 계약 밖 작업이 필요하면 임의 구현하지 않고 결정 필요 항목으로 남깁니다. 구현이 끝나면 메인에게 완료를 보고합니다.

### Phase 3: 검증 및 리뷰 (메인 → Reviewer)

1. **대시보드 최신화:** Generator의 작업이 끝났으므로 대시보드를 [Generator: 대기], [Reviewer: 검증 중]으로 바꿉니다.
2. **검증 요청:** 메인 에이전트가 Reviewer를 호출합니다.
    - 지침: "Generator가 코드를 다 짰다. Sprint2.md의 로그와 소스 코드를 보고, spec.md 규칙에 맞는지 테스트 코드를 짜서 검증해라."
3. **테스트 및 피드백:** Reviewer는 테스트 코드를 작성하고 실행합니다.
    - **성공(Pass) 시:** 메인에게 통과를 보고하고 `Sprint2.md`에 테스트 결과를 남깁니다.
    - **실패(Fail) 시:** 에러 로그를 메인에게 전달합니다. 메인은 다시 대시보드를 [Generator: 수정 중]으로 바꾸고 Generator에게 재작업을 지시합니다.

### Phase 4: 스프린트 종료 및 자산화 (메인 → 개발자)

1. **최종 승인 및 아카이빙:** 모든 테스트가 통과되면 메인 에이전트는 활성 Sprint 문서를 완료 상태로 정리합니다.
2. **지식 베이스 업데이트:** 메인 에이전트는 `Sprint2.md`에 지저분하게 남겨진 서브 에이전트들의 로그를 분석합니다. 그중 개발자가 나중에 반드시 알아야 할 핵심 트러블슈팅 내역(원인, 해결책, 교훈)과 완료 Sprint 요약을 정제하여 `working-memory.md`에 누적합니다. 대시보드 상태는 [모든 에이전트: 대기 - 개발자의 다음 명령 대기 중]으로 변경합니다.
3. **Milestone 상태 갱신:** Sprint 결과로 Milestone의 범위, 순서, 상태가 실제로 바뀐 경우에만 `plan.md`를 갱신합니다.
4. **완료 보고:** 메인 에이전트가 개발자에게 최종 완료를 보고합니다.
- 보고: "스프린트 2의 개발 및 검증이 완료되었습니다. 핵심 트러블슈팅 내역과 현재 현황은 working-memory.md에 정리해 두었으니 확인해 주세요."

## 4. 기대 효과 및 모니터링 방식

- **개발자 중심의 모니터링:** 개발자는 복잡한 코드나 지저분한 서브 에이전트의 로그를 볼 필요 없이 오직 `docs/working-memory.md` 하나만 열어보면 됩니다. 개발 중일 때는 거시적인 대시보드로 현황을 파악하고, 개발이 끝난 후에는 요약된 트러블슈팅 아카이브를 통해 기술적 자산을 취득합니다.
- **컨텍스트 최적화:** AI는 가볍고 구체적인 `SprintN.md`와 `spec.md`만 참조하므로 환각 및 망각 현상이 원천 차단되며 토큰 비용이 획기적으로 절감됩니다.
