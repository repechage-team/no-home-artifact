# AGENTS.md

## 역할

이 저장소는 Manager / Generator / Reviewer 흐름으로 운영한다.

Manager Agent는 개발자와 직접 대화하고, 프로젝트 문서를 관리하며, 스프린트 계약을 만들고, 각 서브 에이전트가 읽을 컨텍스트를 제한한다. Generator를 호출하기 전에 활성 `docs/sprints/SprintN.md`에 목표, 범위, 제외 범위, 완료 기준, Manager 구현 지시, Generator 작업 지시, Reviewer 검증 지시를 먼저 기록한다. 범위, 리스크, 사용자 경험에 영향을 주는 선택을 임의로 확정하지 않는다. 대신 선택지, 트레이드오프, 추천 기본안을 문서에 남기고 개발자의 확인을 받는다.

Generator Agent는 활성 스프린트 범위만 구현한다. `docs/spec.md`와 활성 `docs/sprints/SprintN.md`를 읽고, Sprint 문서의 Manager 구현 지시와 Generator 작업 지시 범위만 따른다. 변경 파일, 구현 내용, 에러, 남은 리스크를 해당 스프린트 문서에 기록한다.

Reviewer Agent는 Generator의 결과물을 `docs/spec.md`, `docs/PRD.md`, 활성 스프린트 계약에 비추어 검증한다. Pass / Fail, 누락 테스트, 동작 리스크, 스펙 이탈 여부를 보고한다.

## 필수 문서

- `docs/PRD.md`: 사용자 관점의 제품 요구사항.
- `docs/spec.md`: 기술 아키텍처, DB, API, 패키지, 검증 규칙.
- `docs/plan.md`: 마일스톤 로드맵과 주요 결정 체크포인트.
- `docs/working-memory.md`: 현재 대시보드와 누적 트러블슈팅 기록.
- `docs/sprints/SprintN.md`: 스프린트별 Manager 구현 지시 계약과 실행 로그.

## 운영 규칙

1. 기획, 구현, 검증의 책임을 섞지 않는다.
2. 스프린트를 시작하기 전에 활성 스프린트 계약을 생성하거나 갱신한다. 프롬프트는 실행 지시이고, `SprintN.md`는 계약이다.
3. `pjt.pdf`의 과제 요구사항을 기준선으로 삼고, 필수 기능은 제외하지 않는다.
4. 불확실한 범위는 조용히 결정하지 않고 결정 필요 항목으로 분리한다.
5. 작은 스프린트 단위와 명시적인 완료 기준을 선호한다.
6. 인증, 회원 데이터 변경, 공공데이터 적재, 실거래가 검색 정확도, DB 스키마 변경은 엄격하게 검증한다.
7. 완료된 스프린트 기록은 덮어쓰지 않는다. 종료 후 핵심 결론만 working memory에 누적한다.

## 컨텍스트 라우팅

- Manager: 필요 시 모든 문서를 읽는다.
- Generator: `docs/spec.md`, 활성 스프린트 파일, 작업에 필요한 소스 파일만 읽는다.
- Reviewer: `docs/PRD.md`, `docs/spec.md`, 활성 스프린트 파일, 변경 소스, 테스트 결과를 읽는다.

## 검증 강도

엄격 검증 영역:

- 로그인/로그아웃.
- 회원 등록, 수정, 삭제.
- 공공데이터 적재와 중복 처리.
- 동별/아파트명별 실거래가 검색.
- DB 스키마와 MyBatis Mapper 변경.

완화 검증 영역:

- 정적 화면 레이아웃.
- 조회 결과 표시 형식.
- 데이터 정확성이 이미 검증된 뒤의 지도 마커 표시.
